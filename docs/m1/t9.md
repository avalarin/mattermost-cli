## Key decisions

### Mode rename: Normal/Command ŌåÆ Input/Messages

`ModeNormal` and `ModeCommand` were replaced with `ModeInput` and `ModeMessages`. The old names were confusing ŌĆö "Normal" was actually a passive state while "Command" was active input. The new names clearly describe what has focus, matching the two-panel mental model (input box vs. message feed).

### textarea vs textinput

Replaced `bubbles/textinput` (single-line) with `bubbles/textarea` (multi-line):
- Allows multi-line message drafts (real chat clients don't restrict to one line)
- Enter key inserts a newline naturally; Alt+Enter is the "send/execute" trigger
- `CharLimit = 4000` (Mattermost's approximate message limit)
- `ShowLineNumbers = false` for a clean look
- `inputHeight = 3` rows ŌĆö compact but allows seeing context

### Why Alt+Enter for send

`Cmd+Enter` cannot be reliably captured in standard terminals (it often sends `Ctrl+Enter` or is swallowed by the terminal emulator). `Alt+Enter` is reliably distinct from plain Enter in most terminal emulators and is a common convention for "send without line break". In bubbletea, it arrives as `tea.KeyMsg{Type: tea.KeyEnter, Alt: true}`, which the key binding `"alt+enter"` matches.

### ctrl+j for FocusMessages (deviation from spec)

The spec specified `ctrl+m` for `FocusMessages`, but `ctrl+m` is ASCII 13 (carriage return), which is identical to Enter (`\r`) in standard terminals. Bubbletea maps both to the string `"enter"`. Using `"ctrl+m"` as a key binding would therefore never fire distinctly from Enter, breaking the textarea input.

**Resolution:** Using `ctrl+j` (ASCII 10, line feed, `keyLF`) which is distinct and reliably bindable. `ctrl+b` for `FocusInput` works as-is since it's ASCII 2 (`keySTX`).

This deviation is documented here and in a code comment. When modern terminal protocols (kitty key protocol) become widely supported by bubbletea, `ctrl+m` can be revisited.

### Double-Esc mechanic

Instead of string-comparing `statusMsg` to detect a repeat Esc press (fragile), a dedicated `escPending bool` field is used. On first Esc: set `escPending = true`, increment `escGen`, show hint, schedule `MsgEscTimeout` after 3s. On second Esc: detect `escPending`, clear input, deselect, reset. On timeout: clear `escPending` and hint if `Gen` still matches. This is safe against stale timers via the generation counter.

### Message cursor implementation (line offsets)

`rerenderFeed()` now builds `msgLineOffsets []int` ŌĆö one entry per `feedItem` containing the starting line number (0-based) of that item in the viewport content. This allows `scrollToSelected()` to call `viewport.SetYOffset(lineOffset)` to jump directly to the selected message. The offsets are computed by counting `\n` characters in each rendered block.

Selection highlight is applied by `highlightBlock()` which wraps each line of a multi-line message block in a lipgloss style with a dark-gray background (`color "237"`), padded to `m.width` so the highlight fills the full terminal width.

### rerenderFeed guards for uninitialized viewport

`rerenderFeed()` is called in many code paths including before the viewport is initialized (`m.ready == false`). The function guards every `m.viewport.SetContent()` call with `if m.ready`, and still builds `m.msgLineOffsets` so that offset data is ready once the viewport is initialized.

## Deviations from spec

| Spec | Actual | Reason |
|------|--------|--------|
| `ctrl+m` for FocusMessages | `ctrl+j` | `ctrl+m == enter` in standard terminals |
| `feedHeight = height - 5 - inputHeight` | same (= `height - 8`) | `inputHeight = 3`, 5 fixed rows |
| Double-Esc exits to clear input/deselect | same, but only in `ModeInput` | Esc in `ModeMessages` immediately goes to input (single press); double-Esc is for clearing input content from `ModeInput` |

## Known limitations

- `ctrl+m` for FocusMessages is impractical in standard terminals; `ctrl+j` is used instead
- `highlightBlock` uses `color "237"` which may not be visible on all terminal themes (light backgrounds)
- `scrollToSelected` uses `SetYOffset` directly ŌĆö if the selected message is taller than the viewport, only the top is visible (no "scroll to keep in view" logic)
- The `textarea` cursor blink command from `Focus()` is discarded in `NewModel()` (safe ŌĆö the blink starts once the tea program is running via the `Cursor.Blink` field)
