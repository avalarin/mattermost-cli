# T7: SQLite + in-memory store (рефактор потока данных)

## What was done

- `internal/store/db.go` — replaced stub with real `database/sql` + `modernc.org/sqlite`; `Open` creates schema, `InsertMessage` (upsert/ON CONFLICT DO NOTHING), `GetRecentMessages`, `GetMessageByID`, `Close`
- `internal/store/store.go` — added `store.Message` type (extends API fields with `SenderName`/`ChannelName` resolved at ingest time), full `Store` implementation: `AddMessage` (cap 1000), `GetParentSnippet` (memory-first, DB fallback), `LoadRecent`, `GetLines`; render logic moved here from `tui`
- `internal/tui/msgs.go` — added `MsgNewMessage` and `MsgHistoryLoaded`
- `internal/tui/model.go` — removed `feedLines []string` and `msgCache map[string]string`; added `store *store.Store`; `Init()` fires one-shot `loadHistory` cmd; `handlePostedEvent` delegates to `store.AddMessage`; `handleWindowSize` init uses `store.GetLines()`; `MsgHistoryLoaded` handler updates viewport when ready; `renderMessageLine` removed (moved to store)
- `internal/tui/model_test.go` — removed three `renderMessageLine` tests (replaced by store tests)
- `cmd/mattermost-cli/main.go` — opens SQLite at `~/.config/mattermost-cli/db.sqlite`, creates store, passes it to `NewModelWithHeader`; handles DB open failure gracefully (runs without persistence rather than crashing)
- New test files: `internal/store/db_test.go`, `internal/store/store_test.go`

## Key decisions

**`store.Message` instead of extending `mattermost.Message`**: The WS event envelope carries `sender_name` and `channel_name` separately from the post body. These fields are needed for rendering and DB persistence. Rather than extending the API type, a dedicated `store.Message` keeps the API layer clean.

**Render function lives in `store`**: `store.AddMessage` returns a rendered display string per spec. Since `tui` imports `store` (not the reverse), rendering must happen inside `store`. The function `renderMessage(msg, snippet)` replaces the old `renderMessageLine(post, senderName, channelName, msgCache)`.

**`Store` maintains `renderedLines []string`**: After removing `feedLines` from `tui.Model`, the store maintains the parallel slice of rendered strings alongside raw messages. `GetLines()` returns a copy. `handlePostedEvent` calls `store.GetLines()` to rebuild the full viewport content after each new message.

**`MsgHistoryLoaded` with lines**: History is loaded in one shot (not per-message). The cmd returns `MsgHistoryLoaded{Lines}`, and the handler updates the viewport if it's already initialized. If viewport initialises after history loads, `handleWindowSize` calls `store.GetLines()` which already has the history.

**Mutex in Store**: `loadHistory` runs in a tea.Cmd goroutine; `handlePostedEvent` runs in the Bubble Tea Update goroutine. Both mutate `messages` and `renderedLines`, so a `sync.Mutex` is required.

**Nil store guard**: `NewModel()` (used in tests) creates a model with no store. All store accesses are guarded with `if m.store != nil` to keep unit tests simple.

**DB open failure is non-fatal in `main.go`**: If SQLite cannot be opened (e.g., filesystem permissions), the app falls back to running without persistence rather than exiting. This matches the spec's intent that the DB is an enhancement, not a hard requirement.

## Deviations from spec

- `LoadRecent` returns `([]string, error)` (rendered lines) rather than `([]Message, error)` (raw messages). The rendered form is what the TUI needs, and the raw messages are already loaded into the in-memory slice as a side-effect. This avoids a second rendering pass in the tea.Cmd lambda.
- Schema WAL mode is set via PRAGMA rather than DSN to avoid URI format ambiguity between file paths and in-memory test DSNs.

## Known limitations

- `feedLines` semantically removed from Model, but a conceptually similar accumulation now lives in `store.renderedLines`. On restart, only the last 100 messages (via `LoadRecent`) are shown, not the entire history.
- `channels` table schema is created but no insert/query methods are provided (not required by T7).
