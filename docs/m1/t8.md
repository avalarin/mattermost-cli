# T8: /send command + DM + command registry

## What was done

- Added `GetUserByUsername(username string) (*User, error)` to `internal/mattermost/client.go`
- Created `internal/tui/command.go` — a generic command registry with `ArgSpec`, `CommandDef`, `Registry`, and `ParseResult` types
- Refactored `internal/tui/model.go`:
  - Replaced `feedMessages []feedMessage` with `feedItems []feedItem` (union of message and system-text items)
  - Added `client *mattermost.Client`, `teamID string`, `registry *Registry` to `Model`
  - Updated `NewModelWithHeader` signature to accept `client` and `teamID`
  - Added `buildRegistry`, `makeSendCmd`, `makeHelpCmd`, `clearStatusAfter` helpers
  - Rewrote `executeCommand` to use the registry; falls back to legacy /quit-only behavior when `registry == nil` (preserves existing `NewModel()` tests)
  - Added `MsgCommandResult`, `MsgClearStatus`, `MsgSystemMessage` handlers in `Update()`
  - Updated `rerenderFeed` to handle both `feedItemKindMessage` and `feedItemKindSystem`
- Added `MsgSystemMessage` to `internal/tui/msgs.go`
- Updated `cmd/mattermost-cli/main.go`: `loadStartupState` now also returns `*mattermost.Client` and `teamID string`; these are passed to `NewModelWithHeader`
- New test files: `internal/tui/command_test.go`, tests added to `internal/mattermost/client_test.go` and `internal/tui/model_test.go`

## Key decisions

**Command registry over a switch statement.** The spec described a simple `parseCommand` function, but a registry pattern was chosen (as specified in the implementation instructions) because it keeps each command self-contained, makes /help auto-documenting from `ArgSpec`, and makes adding future commands trivial (no model.go changes needed, just register a new `CommandDef`).

**Greedy last-arg for /send text.** The `text` argument uses `Greedy: true` so `/send #general hello world` captures "hello world" as a single string without requiring quoting. This matches the intended UX.

**feedItem union type.** The /help command injects a system message directly into the feed. A `feedItemKind` enum with `feedItemKindMessage` / `feedItemKindSystem` was added instead of storing a rendered string in `feedMessage`, preserving the invariant that message items always re-render on resize while system items are inserted verbatim.

**registry == nil fallback.** `NewModel()` (no args) is used in existing TUI tests that don't need a registry. Rather than changing those tests, `executeCommand` degrades gracefully when `m.registry == nil` — it only recognizes `/quit`, mirroring the pre-T8 behavior.

**clearStatusAfter uses time.Sleep.** A blocking `tea.Cmd` is idiomatic in Bubble Tea for simple timers. The 2-second delay runs on a goroutine managed by the runtime, not the main loop.

## Deviations from spec

The spec mentioned a `parseCommand` function returning a `Command` interface (`SendCommand`, `QuitCommand`). The implementation uses the registry pattern instead, which subsumes that design. The behavior and error messages are equivalent.

The `ErrMissingArg` message includes the usage string inline (e.g., `missing argument: target — usage: /send <target> <text>`) rather than requiring `executeCommand` to look up the definition separately. This is slightly more informative.

## Known limitations

- `/send` requires an active REST client; if `client == nil` (e.g., config missing), it returns "not connected" immediately.
- The team ID passed to `/send` is fixed at startup — if the user switches teams within a session (not a current feature), the DM lookup would use the wrong team context.
- `/help` output is appended to the feed as a plain string, so it re-renders verbatim on resize (no word-wrap). This is acceptable for the short help text produced.
