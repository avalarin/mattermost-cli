# T3: Config package

## What was done

- `internal/config/config.go`: full `Load(path string) (*Config, error)` implementation
  - Reads TOML via `github.com/BurntSushi/toml`
  - Applies defaults before parsing (so TOML values override defaults)
  - Applies env overrides after parsing: `MATTERMOST_URL`, `MATTERMOST_TOKEN`, `ANTHROPIC_API_KEY`
  - Validates required fields (`server.url`, `server.token`); returns `ErrMissingRequiredField` sentinel
- `internal/config/config_test.go`: 6 tests covering all spec scenarios plus missing-file case
- `cmd/mattermost-cli/main.go`: wired to `config.Load()` via `loadConfigStatus()`, passes result to TUI status bar
- `internal/tui/model.go`: added `NewModelWithStatus(status string) Model` constructor
- `Makefile` and `Justfile`: `build` target changed to `go build -o mattermost-cli ./cmd/mattermost-cli/` to produce a binary in the project root
- `Justfile`: added `just dev` recipe — runs the app with `config.dev.toml`

## Key decisions

**Missing file is not fatal**: If the config file doesn't exist, `Load` skips TOML parsing and falls through to validation. Validation then fails with `ErrMissingRequiredField` (since URL and Token are empty). The status message shown in the TUI is "Config file not found. Run with --config..." — friendly, no panic.

**Defaults applied before TOML decode**: `applyDefaults` runs first so that any field present in the TOML file overwrites the default, and absent fields keep the default. This is the standard pattern with `BurntSushi/toml`.

**`ErrMissingRequiredField` sentinel**: Wrapping with `fmt.Errorf("%w: field.name", ErrMissingRequiredField)` lets callers use `errors.Is` to distinguish config validation errors from I/O or parse errors.

**`NewModelWithStatus` constructor**: Kept `NewModel()` signature intact (tests depend on it) and added `NewModelWithStatus` as a thin wrapper. Avoids functional options boilerplate for a single use case.

## Deviations from spec

None — all five spec tests implemented, env override and validation match spec exactly.

## Known limitations

- `config.Load` does not validate `server.url` format (URL syntax) — treated as an opaque string. Validation will happen implicitly when the HTTP client tries to connect in T4.
- `server.team` is not validated as required — the spec doesn't list it as a required field, though the app needs it for T4.
