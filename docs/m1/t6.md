## Key decisions

**coder/websocket instead of nhooyr.io/websocket**
The spec called for nhooyr.io/websocket, but staticcheck flagged it as deprecated — the library was forked and is now maintained at github.com/coder/websocket. The API is identical, so the switch was drop-in.

**ConnStatus moved to mattermost package**
Moving it there keeps the mattermost package self-contained (WSClient produces statuses, no import cycle) and lets main.go use mattermost.ConnStatusConnecting without importing tui.

**Raw mattermost.Event flows into model.Update()**
Rather than decoding events in a separate goroutine into MsgNewMessage, the raw Event is returned from waitForEvent and decoded in handlePostedEvent inside model.Update. This keeps all state mutation on the Bubble Tea event loop and avoids race conditions.

**sender_name @ prefix stripping**
Mattermost sends sender_name as "@username". The @ is stripped before display so the rendered line reads "alice: text" not "@alice: text".

**atBottom defaults to true**
Auto-scroll is on by default. Any upward key scroll (detected by comparing ScrollPercent before/after) sets atBottom=false. The End key restores it.

**GetChannelsForTeam fetch at startup**
Channel names are fetched once at startup and passed into the model as a lookup map. This avoids any async REST calls during the live WS feed.

## Deviations from spec

- `MsgNewMessage.SenderName` and `MsgNewMessage.ChannelName` fields exist in msgs.go but are not used yet — the event decoding happens directly in handlePostedEvent. They are kept for T7 (SQLite store) when they will be needed.
- loadStartupState now also calls GetChannelsForTeam; a failure there is non-fatal (logged at debug level, channels map stays empty — channel IDs show instead of names).

## Known limitations

- Thread replies can only show a parent snippet if the parent post arrived in the same session (msgCache is in-memory). T7 (SQLite store) will persist posts across restarts.
- No deduplication — if the same event is received twice, a duplicate line is appended.
- The WS connection uses the same token as the REST client (personal access token). Session token auth is not supported.
