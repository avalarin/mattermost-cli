# T4: Mattermost types + REST client

## What was done

- Added JSON struct tags to `internal/mattermost/types.go` to map Mattermost API v4 field names (`id`, `channel_id`, `user_id`, `message`, `create_at`, `root_id`)
- Added `EventTypePosted` constant
- Implemented all REST methods in `internal/mattermost/client.go` using `net/http`
- Added exported sentinel errors: `ErrChannelNotFound`, `ErrAuthFailed`
- Added `ConnStatus` and `HeaderInfo` types to `internal/tui/model.go` and updated `renderHeader()` to show `mattermost-cli  [connecting]  team: <name>  @<user>`
- Added `NewModelWithHeader()` constructor to TUI
- Updated `cmd/mattermost-cli/main.go` to authenticate at startup; exits with code 1 on config validation error or auth failure
- Added `internal/mattermost/client_test.go` with 6 tests using `net/http/httptest` mock servers

## Key decisions

- **`errNotFound` unexported sentinel**: The `get` helper returns an unexported `errNotFound` on HTTP 404; callers wrap it with domain-specific exported errors (e.g., `ErrChannelNotFound`). This keeps the public API clean.
- **`currentUserID` caching in Client**: `GetCurrentUser()` caches the user ID for reuse by `FindOrCreateDM`, avoiding a second API call. Thread-safety is not a concern in M1 (single goroutine init).
- **`FindOrCreateDM` signature keeps `teamID`**: The spec defines this signature. The teamID is accepted but not used by the direct channel API — kept for interface consistency with future tasks.
- **Missing config file is not exit(1)**: Only config validation errors and auth failures cause exit(1). A missing config file starts the TUI with a helpful message, keeping the app launchable without configuration.

## Deviations from spec

None.

## Known limitations

- `GetChannelsForTeam` uses `/users/me/teams/{teamID}/channels` — this returns only channels the current user is a member of. Full team channel listing would require `/teams/{teamID}/channels`.
- No request timeout configured on the HTTP client (acceptable for M1).
