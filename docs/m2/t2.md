## Key decisions

**PostList response structure:** The Mattermost v4 API returns `{order: [...newest-first], posts: {id: msg}}`. Walking `order` in reverse builds a chronological slice without any sorting step.

**`getQ` helper:** A separate `getQ(path, query, out)` method was added rather than changing the existing `get` method signature. The `query` param is caller-controlled and safe for integer values; future callers with string values must use `url.QueryEscape`.

**Stale result discard:** `MsgChannelHistory` handler checks `msg.ChannelID != m.activeChannelID` and discards results if the user switched channels while the fetch was in flight.

**Page tracking in model, not store:** `activePage map[string]int` lives on the model because it is pagination UI state. The store only holds the message data. Initial load sets `activePage[channelID] = 1`; each prepend load increments it.

**WS message filtering in `handlePostedEvent`:** When a specific channel is active, only messages from that channel are added to `messagesView`. Messages from other channels still go to the global store but don't pollute the filtered view.

**Infinite scroll guard:** The Up/PageUp triggers check `!m.messagesView.IsEmpty()` to avoid firing a load when the viewport is empty (YOffset==0 is vacuously true on empty views).

**Cap direction:** `AddChannelMessages` with `prepend=true` keeps `combined[:cap]` (oldest messages) when the cap is exceeded, so older pages loaded by infinite scroll survive. `prepend=false` keeps `combined[len-cap:]` (newest).

## Deviations from spec

- `SenderName` for REST-loaded history is set to `p.UserID` (not the display username). This is a known limitation for T2; username resolution via the `sender_name` WS envelope works for live messages but REST `GetChannelPosts` does not return usernames.

## Known limitations

- Username resolution for history messages shows raw user IDs until a dedicated resolution pass is implemented.
