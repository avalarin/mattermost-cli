# M2 T3 — Только root-сообщения + бейдж ответов

## What was done

- **`internal/config/config.go`**: added `ChannelMessages string` (`toml:"channel_messages"`) to `UIConfig` with default `"root_only"`.
- **`internal/store/store.go`**: added `ReplyCount int` to `Message` struct; added `IncrementReplyCount(rootID string)` that updates both the global in-memory list, all per-channel caches, and the DB (best-effort).
- **`internal/store/db.go`**: added `reply_count INTEGER NOT NULL DEFAULT 0` to the `CREATE TABLE messages` DDL; added `ALTER TABLE messages ADD COLUMN reply_count ...` migration for existing databases (error ignored if column already exists); updated `InsertMessage`, `GetRecentMessages`, `GetMessageByID` to include the new column; added `IncrementReplyCount(id string) error`.
- **`internal/tui/messages_view.go`**: added `isAllActivity bool` field to `MessagesView`; added `SetAllActivity(bool) MessagesView` and `IncrementReplyCount(rootID string) MessagesView`; updated `renderMessageLine` to accept `isAllActivity bool` and append `⤵︎ N` for root messages with replies, `⤴︎` for replies shown in All Activity.
- **`internal/tui/model.go`**: added `channelMessages string` field; added `channelMessages` param to `NewModelWithHeader`; in `MsgChannelSelected`, sets `isAllActivity` on the view; in `MsgChannelHistory`, propagates `ReplyCount` from REST messages and filters out replies when `root_only`; in `handlePostedEvent`, calls `store.IncrementReplyCount` + `messagesView.IncrementReplyCount` when a reply arrives, and skips adding replies to the feed in root_only channel mode.
- **`cmd/mattermost-cli/main.go`**: updated `loadStartupState` return signature and `NewModelWithHeader` call to pass `cfg.UI.ChannelMessages`.

## Key decisions

- **Filtering at model level, not in MessagesView**: replies are excluded from `feedItems` when building items for a channel in `root_only` mode. This keeps `MessagesView` simple (no offset/selection logic for hidden items).
- **`isAllActivity` in MessagesView for badge rendering only**: the view doesn't filter — it just uses the flag to decide whether to show `⤴︎` on replies.
- **`IncrementReplyCount` outside the `if m.store != nil` block**: the messagesView badge update happens unconditionally so tests without a store still work.
- **DB migration via `ALTER TABLE`**: swallowing the "duplicate column" error is the standard SQLite migration pattern — no separate migration table needed for a single column addition.

## Deviations from spec

None.

## Known limitations

- `ReplyCount` in WS `posted` events reflects only the new message's own reply count (always 0 for a fresh post). The parent's count is incremented by 1 per event, which is correct for live events. History loaded via REST uses the accurate `reply_count` from the API response.
- The `⤴︎` badge appears in All Activity next to the `↩` reply indicator — both show on the same header line. This is intentional per spec ("badge `⤴︎`" in addition to the existing reply indicator).
