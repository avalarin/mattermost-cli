## Key decisions

**Value receivers throughout** — both `MessagesView` and `ChannelsView` use value receivers returning the new value. This is consistent with Bubble Tea's immutable update model and keeps the mutation pattern uniform with `Model` itself.

**MessagesView.SetSize reserves one header line** — viewport height is `feedH - 1`; the caller (`renderBody`) renders the messages header separately and places it above the viewport. This keeps the header bold/styled text independent of the scrollable viewport content.

**ChannelsView.View pads to full height** — after rendering visible items, empty lines are appended so the panel always occupies exactly `feedH` rows. This prevents the side-by-side zip in `renderBody` from producing mismatched line counts.

**loadStartupState returns channelsWidth** — instead of threading `*config.Config` through `main()`, the function returns the single value needed. Avoids exposing the full config struct to the caller.

**rerenderFeed kept package-private** — `MessagesView.rerenderFeed` is unexported. `model.go` calls it directly only inside `handleKeyMessages` for the End key (where a rerender is needed after SelectLast + GotoBottom). All other mutations go through the public methods that already call rerenderFeed internally.

## Deviations from spec

- `MsgChannelSelected` handler in `Update` calls `SetOpenByID` on `channelsView` (not specified in the spec's T1 section, but required to keep openIdx in sync when the message is processed).
- The `NewModelWithHeader` signature adds `channelsWidth int` as the last parameter (the spec says "add `channelsWidth int` param after `teamID string`").

## Post-review fixes

After code review the following warnings were addressed:

- `SelectedItem()` now returns `(feedItem, bool)` instead of `*feedItem` to avoid the pointer-to-copy footgun (the old `*feedItem` pointed at a local copy, not into the slice).
- `feedH` is stored on `Model` and updated only in `handleWindowSize`/`syncInputHeight`; `renderBody` reads `m.feedH` instead of re-deriving it from `m.input.LineCount()`, eliminating a potential off-by-one when the textarea grows mid-frame.
- Added `ChannelsView.DisplayNameByID(channelID)` so `renderMessagesHeader` doesn't reach into `channelsView.items` directly.
- Fixed stale test error string ("ctrl+m" → "ctrl+j") in `model_test.go`.
- Fixed `wrapText` doc comment (blank lines are dropped, not preserved).

## Known limitations

- No actual channel filtering yet — switching channels updates `activeChannelID` and the header label but the messages feed still shows all messages (T2 scope).
- `ChannelsView` items are only loaded once at startup; newly joined channels don't appear without restart (T2/T3 scope).
