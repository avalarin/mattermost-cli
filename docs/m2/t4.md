## Key decisions

**Value-receiver pattern for ThreadPopup**: mirrors `MessagesView` exactly — all mutation methods return a new value, and `Model` holds a `*ThreadPopup` pointer (nil = closed). Updating via `tp := m.threadPopup.Method(); m.threadPopup = &tp`.

**`fullText` parameter on `renderMessageLine`**: chosen over a separate function to keep the rendering logic in one place. The thread popup always passes `true`; the main feed always passes `false`.

**`/reply` intercepted before registry dispatch**: the registry `/reply` registration is a thin stub (so `/help` and `/help reply` work) but the real behavior is handled in `executeCommand` before `registry.Parse()` is called. This avoids the registry needing to know about model state.

**`replyRootID` injection between Parse and Execute**: after registry parses `/send`, we check `replyRootID` and swap in `makeSendReplyCmd` (which sends with `rootID`). This keeps the registry `/send` definition unmodified.

**Thread popup dimensions**: width `min(80, total-4)` (capped at 80), height `feedH` (same as message panel). This fills the feed area without obscuring the header/status/input.

**`MsgThreadLoaded.UserNames`** carries a map from the batch-resolved usernames so the popup can display them immediately without an additional round-trip.

## Deviations from spec

- Thread popup is in `internal/tui/thread_popup.go` (same package, not `tui/views/`) as instructed by the task spec override — the task explicitly placed it in `internal/tui/`.
- `r` key sets `/reply` in the input and switches to `ModeInput` (popup stays open); actual rootID is stored in `openThreadRootID` and consumed when the user executes `/send`.

## Known limitations

- `e` (edit) and `d` (delete) are stubbed: edit inserts `/edit <text>` into input (command not yet implemented), delete shows "not implemented yet" in the status bar.
- `/reply` sets context for the *next* `/send`; if the user types something else first, the reply context is consumed anyway (by the `/send` command).
